Setup (one sentence)
- Five-node leader–follower and five-node leaderless stacks on one EC2 host behind an Elastic IP; load generators ran from a laptop against public HTTP ports; LF quorum was switched across w5r1 / w1r5 / r3w3 via SSM editing docker-compose on the instance.

Throughput vs write fraction (leader–follower)
- w5r1 (W=5,R=1): throughput drops sharply as write ratio rises (1421 → 319 ops in 50s) because every write pays for synchronized replication to all followers before ack.
- w1r5 (W=1,R=5): reads dominate cost (parallel fan-out + simulated fetch delay); writes return quickly after the leader commit, so at high write ratios aggregate ops can exceed w5r1 (2160 ops at 90% writes vs 319 for w5r1).
- r3w3: sits between the extremes; writes are cheaper than w5r1 but reads still do a three-way merge, so curves reflect a balanced quorum.

Staleness
- Leader–follower: 0 stale reads in this AWS run for all three profiles and all write ratios—reads go through the leader’s quorum/merge path as implemented.
- Leaderless: R=1 local reads allow temporary lag behind the writer’s replica; observed stale reads at low write mix (6 at 1% writes, 4 at 10%) when reads are frequent relative to completed replication; 0 stale at 50% and 90% writes where the workload behaves more write-heavy or replication catches up within the client’s hot-key window.
