Setup (one sentence)
- Five-node leader–follower and five-node leaderless stacks on one EC2 host behind an Elastic IP; load generators ran from a laptop against public HTTP ports; LF quorum was switched across w5r1 / w1r5 / r3w3 via SSM editing docker-compose on the instance.

Which leader–follower profile “wins” on throughput (ops in 50s) per write ratio?
Use your JSON if you re-run; this is the AWS sweep summarized in AWS_RUN_SUMMARY.txt:
- **1% writes:** **w5r1** (1421) > **r3w3** (1273) > **w1r5** (843). Reads dominate; **R=1** (leader-local) avoids read fan-out, so **w5r1** leads. **w1r5** pays **R=5** on every `cluster-read`, so it is slowest.
- **10% writes:** **w1r5** (1538) > **r3w3** (1259) > **w5r1** (1221). Even with mostly reads, **w1r5** wins on *count of completed ops* here: cheap leader commits on the minority writes avoid the long **W=5** write path that **w5r1** pays whenever a write happens; the mix shifts total work enough that leader latency dominates. (Intuition: r3w3 stays in the middle; w5r1 pays full replica sync on each write.)
- **50% writes:** **w1r5** (1825) >> **r3w3** (797) > **w5r1** (536). **Cheap write ack (W=1)** dominates when half the ops are writes.
- **90% writes:** **w1r5** (2160) >> **r3w3** (549) > **w5r1** (319). **w5r1** collapses because every write waits for all followers synchronously.

Reasoning (tie to mechanism, not graph shapes)
- **w5r1** minimizes **read path cost** (R=1) but maximizes **write path cost** (W=5) — best when writes are **rare**.
- **w1r5** minimizes **leader write return time** (W=1) but pays **R=5** read fan-out — still wins at **high write fractions** in this experiment because synchronous **W=5** on w5r1 is the dominant bottleneck once writes are common.
- **r3w3** balances **W=3** and **R=3** — typically **middle** throughput; good when you want a compromise between the two extremes.

Throughput vs write fraction (leader–follower)
- w5r1 (W=5,R=1): throughput drops sharply as write ratio rises (1421 → 319 ops in 50s) because every write pays for synchronized replication to all followers before ack.
- w1r5 (W=1,R=5): reads dominate cost (parallel fan-out + simulated fetch delay); writes return quickly after the leader commit, so at high write ratios aggregate ops can exceed w5r1 (2160 ops at 90% writes vs 319 for w5r1).
- r3w3: sits between the extremes; writes are cheaper than w5r1 but reads still do a three-way merge, so curves reflect a balanced quorum.

Staleness
- Leader–follower: 0 stale reads in this AWS run for all three profiles and all write ratios—reads go through the leader’s quorum/merge path as implemented.
- Leaderless: R=1 local reads allow temporary lag behind the writer’s replica; observed stale reads at low write mix (6 at 1% writes, 4 at 10%) when reads are frequent relative to completed replication; 0 stale at 50% and 90% writes where the workload behaves more write-heavy or replication catches up within the client’s hot-key window.

Application fit (which database style for what workload?)
- **w5r1 (strong writes, cheap reads):** Workloads with **infrequent writes** and a need for **linearizable-or-strong reads** without fan-out — e.g. rarely-updated reference data, configuration, or “read-mostly” catalogs where write latency is acceptable.
- **w1r5 (cheap writes, expensive quorum reads):** **Write-heavy** or ingest pipelines where **low write latency** matters and you can pay for **read reconciliation** (analytics, logging metadata, dashboards that can tolerate slower reads or run off-peak).
- **r3w3:** **Balanced** OLTP-style mixes where you want **better availability / latency tradeoffs** than all-replica writes but **stronger** reads than a single node — typical “middle quorum” for general services.
- **Leaderless W=N, R=1:** **Low read latency**, **partition-tolerant scale-out** memcached/dynamo-flavored use cases where **eventual consistency** and occasional **stale reads** are acceptable (feature flags, social counters, session caches). **Not** for balances or inventory without application-level versioning.
